## 1. 溢出处理：本轮最硬的探索（根治"硬撑")

### 痛点（实测基线）
设备内容区 CardStage 高度固定 **273px**（`375 − top36 − bottom66`；*这是本轮基线，第三轮放大到 **287**：top30/bottom58，见 §7*）。改造前组件 `flex:0 0 auto` 按内容撑开、舞台 `overflow-y:auto` —— 内容一多，**卡片整体被撑高、舞台滚动、卡头被滚走、字幕被压**。
用 `_review/_audit/measure.js` 对 9 场景灌入"压力内容"实测（densify：多塞 6 行 / 4 段 / 5 条 + 超长文案）：

| 场景/皮肤 | 最高卡高(改造前) | 溢出 CardStage | 压字幕 |
|---|---|---|---|
| 邮件 glass | 750px | **+477** | 226px |
| 邮件×日程 glass | 750px | +477 | 236px |
| 新闻 bubble | 695px | +422 | 199px |
| 天气 glass | 643px | +370 | 173px |
| 旅行/会议 glass | 568px | +295 | 135px |

→ 卡片胀到舞台的 2–3 倍，严重压住字幕。**铁证：尺寸不固定就一定崩。**

### 探索的三条路 → 择优
| 方向 | 试法 | 取舍 |
|---|---|---|
| A. 整卡缩放（scale 卡片塞进 CardStage） | 内容多→卡整体缩小 | **否决**：字会糊、不同卡尺寸跳动、观感差 |
| B. 卡头脚常驻 + 正文内部滚动窗口 | `pop-large` `max-height:100%`，`pl-head/divider/pl-footer` 固定，`pl-body` `flex:1;overflow-y:auto` + 渐隐边 + TTS 高亮 `scrollIntoView` | **胜出**：核心(头/脚/结论)永在，细节滚动，符合 Figma 真值(查消息卡=固定头+滚动内容) |
| C. 一律降级裁剪 | 全靠 `-webkit-line-clamp` | **部分采用**：单字段过长用 clamp，但多条目硬裁会丢信息，故仅作兜底 |

**定稿（写进 UI-SPEC §0/§9）**：每类组件按主策略分工 —— ListCard/SectionCard/SubjectCard/气泡串=**内部滚动(a)**；新闻 list↔focus=**换形式(b)**；确认门/单字段=**裁剪(c)**；沉浸封面 158×224(list)/254(focus) < 287 天然安全。

### 改造后实测（同一压力内容）
全 16 组合（含内置变体）`maxOverflowPx=0`、`maxSubOverlapPx=0`、超高处 `pl-body` 均有滚动。**零溢出零重叠达成**，且既有紧凑卡视觉零回退（对比 `r_travel_glass.png` ≈ `b_travel_glass.png`）。

## 2. 字幕 1 行 vs 2 行：与卡的避让探索
- 痛点：2 行字幕（~60px，顶边距底 ~74px）会探进 CardStage 底（66px），压住高结果卡。
- 试 1：全局 1 行 → **否决**：陪聊纯语音的长共情句被截断（"…先别审判自己，缓一下" 被切）。
- 试 2：全局 2 行 → 否决：结果卡阶段压卡。
- **胜出**：上下文相关 —— **结果卡在场 → 1 行**（详情留卡）；**澄清/确认/纯语音 → 2 行**（无高卡可压）。引擎按当前活动卡切 `.sub-1line`（结果卡加，澄清/清场不加）。
- 配套：口播文案在结果步压到 ≤约 21 全角字（见 §4），1 行不省略号；静帧隐藏闪烁光标（截图不露馅）。

## 3. 控制台对比视图：实现路径探索（先调研后落地）
后台 agent 跑 `ui-ux-pro-max` + Web 调研"flow builder / 并排 diff / 契约 chips"，要点：稳定 id 选择、撤销栈、toast 撤销、LCS 对齐、契约 chips 按 `frame→tool→evidence→risk→confirm` 顺序读、违规态高亮。
- 变体对比**双活引擎**（两台 device 同时真播）：**否决** —— 引擎是单例、改造面大、/browse 单例难双播。
- **胜出**：静态 hero 预览（`LoonaUI.build` 直接渲该变体结果卡进 mini-device）+ 事件链 chips + **LCS 结构对齐 diff**（相同/改动/独有）+ **契约体检表**（最高风险/确认门数/低证据/工具/步数）作择优依据。轻、零依赖、信息够。
- 额外采纳：**实时契约校验**（spec 不变量 `action_risk≥R3 ⇒ 必须确认门`）——链路里有 R3+ 动作却无确认门 → 该步 `⚠` 高亮。删掉确认门→2 处 R3 立刻报警（实测）。这把"控制台"升级成"会抓违规的审计台"。

## 4. TTS persona：以职业视角择优（反思 agent 按职业审稿）
反思 agent 以"主播/规划师/天气主播/会议秘书/助理/协调者/食客朋友/先接住情绪的人"视角逐句审。胜出原则：**口播给判断/结论/下一步，字段细节留卡**；结果步 ≤约 21 全角字。
固化的几处典型取舍：
- 新闻：`第一条/第二条…`（念行号）→ **念要点**（`大模型上下文翻倍到两百万，长文一次喂得下`）——高亮已指示是哪条。
- 日程：`后面卡在两段：十四点评审、十四点半同步，中间只有十分钟`(28字) → `评审和同步只隔十分钟，太挤了`(时刻在卡上)。
- 餐厅：`直接给你定一家`（"定"=订位 R3 误导）→ `给你挑一家`；主推句砍掉卡上已有的距离/对比。
- 旅行：日句保留"体力读数"（最轻松/节奏正常），砍重复。
- 邮件×日程：开场长铺陈 → `先看日历放不放，回信确认了再发`（两件事在卡的步骤行里）。
- 陪聊 / 会议：**原样保留**（反思 5/5：先接住情绪不给方法论；先决策再风险再下一步、负责人 pending 不乱填、收件人不自动推断）。

## 7. 第三轮增量（失败演示 / persona 拉平 / 控制台加深 / 校验跑满）

补齐 5 个优化点（全部 /browse 实测 + 截图验证）：

1. **失败 / 部分成功演示**（最大空白 → 补上）。规格反复强调"诚实报状态、谁知道/谁不知道、永远给下一步"(REQ-FAIL-001)，但 9 条链曾全是成功路径。
   - 取舍：失败该走**确认门之后**（R3 外发须先确认才尝试），而非凭空失败。故给引擎 `_onConfirmAction` 加 `content.outcome:"fail"` 分支：确认 → `发送中 → 发送失败`，**不留"已发送·查看详情"假回执**，下一步交给随后的 FailureCard。改动 8 行、可复用。
   - 做成**内置变体**而非改基准：邮件×日程 拆成 `OPEN`(共用前半段) + `SUCCESS_TAIL` / `FAIL_TAIL`，base=全成功、variants[0]=部分失败。并排对比时 LCS 只在第 2 道门后标"改动/独有"，叙事清晰。
   - FailureCard：✓日程已改 / ✕邮件没发出·小周还不知道改期 / reason / 下一步三选项；TTS 多拍诚实播报（每拍 ≤21 字，结果卡 1 行不截断）。`measure.js` 变体 sweep 实测该卡最高 261px、0 溢出。
2. **persona 深度拉平**：天气(主播)/邮件(助理)/餐厅(店长)/会议(参谋) 从 1–2 拍提到与新闻/旅行同档的 **4–5 拍**。坑：结果卡恒 `sub-1line`，故每拍**限 ≤21 全角字**（截图 `persona_weather_beat.png` 确认 1 行不糊）。TTS 是字幕，无 DOM，0 溢出风险。
3. **控制台加深**：①对比页底「评审结论」条（更倾向 A/B/势均力敌 + 备注，localStorage 持久、随导出带走）→ 对比直接产出团队结论；②导出补 `annotations_resolved`（自描述：带事件号/类型/原文）+ `review`(verdict)；③`validateStep` 加两条契约校验——**私域数据未声明授权**(read_private_data 全链须 OAuth/已连) + **外发收件人未明确**。坑：会不会误伤理想链？→ 给 calendar/workflow 的 ROUTER 补 `OAuth已连` 声明，9 条理想链体检表全绿，校验只在**编辑出违规时**亮 ⚠。
4. **§12 校验跑满全矩阵**：`verify.js` 之前 token/宽度只抽查邮件/新闻聚焦/澄清；重写成与 `measure.js` 同矩阵——9 case×形态逐组合断言 token/宽度/溢出/字幕。结果 **11/11 PASS**（4 标准卡@430+token、新闻聚焦 548、8 确认门按钮 h32/r999、13 澄清/确认字幕 2 行安全）。`measure.js` 加变体 sweep，全 16 组合(含 FailureCard/问城市版) normal+stress 仍 0/0。
5. **文档同步**：README(内置变体机制 + 新 comp + 契约校验 + 评审结论)、CARD_TAXONOMY(§6 交互模式字段速查)、UI-SPEC(§14.7 失败模式 + 不变量 #10 + §12 #10 全矩阵 + persona #7) + 本页 + 对照页 ⑦ 节。
